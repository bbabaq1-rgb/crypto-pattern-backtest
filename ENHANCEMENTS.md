# 대시보드·자동매매 고도화 검토 (2026-07-06)

## 1. UNI "매도됐는데 오픈으로 보임" — 원인·수정

**결론: 데이터는 정상, 표시(라벨)가 오해를 유발했다. 실거래는 매도되지 않았다.**

- OKX 실측: UNI 롱 진입 2.866, 현재 **오픈**(미실현 +$3.80). → 오픈포지션 표시는 옳다.
- 매매기록의 "UNI +10.6% tp"는 **방식A(±10%) 페이퍼 시뮬레이션** 결과다.
  방식A는 실거래 매도 조건이 아니다(실거래 매도 = 방식D: -8%손절/반대신호/레짐전환/30봉).
- 버그: 방식A 청산이 live 포지션이라는 이유로 `·실거래` 마커를 상속받아,
  실거래 탭에 "매도"로 표시됨 → 실제로는 D가 홀딩 중이라 오픈으로 남음 → 불일치처럼 보임.

**수정**
- `_record_trade`: 방식A는 항상 페이퍼로 기록(live_mode=False). 방식D만 실거래.
- 기존 오라벨 3건(UNI·COMP·DYDX 방식A) DB에서 `·실거래` 마커 제거 → 페이퍼 탭으로 이동.
- 이제 실거래 탭엔 UNI가 오픈으로만 보이고(실제와 일치), +10.6%는 페이퍼 탭에 표시.

**매도 시스템 상태: 정상 작동.** 방식D 청산은 실제 OKX reduceOnly 주문과 연결돼 있고
(커밋 af7b3c4), 수동청산 버튼도 실주문+기록이 검증됨(DYDX +1.56 USDT). UNI는 아직
D 조건 미충족이라 홀딩 중일 뿐이다.

## 2. 발견된 자동매매 사각지대 — 수정

**OKX 13종 실거래 중 7종이 DB에 페이퍼(AD)로 잘못 기록 → 방식D 자동청산 사각지대.**
(ARKM·GLM·ICX·NEO·OP·PENDLE·SAND — 실제 OKX 포지션인데 D-모니터가 페이퍼로 간주해
실주문을 안 냄. 진입경로/러너 복원에서 live 표기 유실이 원인.)

**수정**
- `reconcile_live_flag()`: 스케줄러 매 실행마다 OKX 실측과 대조해, OKX에 실재하는
  포지션을 live_mode=True로 승격 → 방식D 모니터가 실거래로 관리(실주문). entry_date가
  캔들에 안 맞으면 D-eval이 안전 스킵되므로 잘못된 청산 위험 없음.
- 일회성 DB 보정: 해당 9행(중복 포함)을 `method='AD-LIVE'`로 승격 완료.

**부수 효과(안전 강화): MAX_LIVE_POS(5) 실효화.** 그간 live 카운트 누락으로 동시
포지션 제한이 무력화돼 13종까지 늘었다. 이제 live_mode가 정확해져 제한이 실제로 걸린다
→ **포지션이 5개 밑으로 줄기 전까지 신규 실거래 진입이 자동 보류된다**(과레버리지 방지).
현재 $291 자산에 13포지션·가용 $62 상태라 보수적으로 진입을 멈추는 게 맞다.

## 3. 오픈 포지션 표시 — 바이낸스식 개편

- **표시 원천을 OKX 실측으로 고정.** DB(positions)는 러너 소멸·중복으로 실제와
  어긋날 수 있어(31행 vs OKX 13), 실거래 탭은 항상 OKX API를 신뢰.
- 그리드: **종목 · 방향뱃지 · 레버리지뱃지(2x)** / 수량 / 진입가 / 현재가 / **청산가** /
  증거금 / **손익(ROE%)** / 청산버튼. (요구대로 레버리지 정정 표기, 스탑로스 설정란 없음)
- 청산가는 OKX liqPx, 없으면 격리마진 근사. ROE=미실현/증거금.

## 4. 새로고침 방식 — 전체 rerun → 부분갱신(폰트 페이드 제거)

**기존:** `time.sleep(30); st.rerun()` — 30초마다 전체 스크립트 재실행 →
페이지 전체 재렌더(폰트가 옅어졌다 진해지는 깜빡임) + 초기 로딩 시 블로킹.

**개선:** `@st.fragment(run_every="30s")`로 **실시간부(잔고·미실현·오픈포지션)만
부분 갱신**. 바이낸스처럼 숫자만 바뀌고 나머지(매매내역·차트)는 재렌더 안 함.
전체 페이지 깜빡임 제거 + 블로킹 sleep 제거로 초기 표시 빨라짐.

## 5. 모바일 초기 로딩 속도

- 블로킹 `sleep(30)` 제거로 셸이 즉시 렌더. 무거운 데이터(Supabase×3, OKX, 시세)는
  캐시(ttl=25s)로 재방문 시 즉시. 정적 섹션(차트)은 1회만 로드.
- 남은 개선여지: 최초 콜드 로드 시 OKX/Supabase 왕복이 순차 → 향후 병렬화 여지(문서화만).

## 6. React(Next.js) 전환 검토 — 결론: 현 단계 불필요

| 항목 | Streamlit(현재+fragment) | Next.js 전환 |
|---|---|---|
| 부분 숫자 갱신 | ✅ fragment로 확보 | ✅ (websocket/polling) |
| 초기 로딩 | 캐시로 개선 | 더 빠름(정적+CSR) |
| 구축 비용 | 낮음(현 코드 유지) | **높음** — Supabase/OKX용 백엔드 API,
인증, 차트 React 재구현, 호스팅 이전(Vercel 등) 다일 작업 |
| 배포 | GitHub push→자동 | 파이프라인 재구성 |

**권장:** fragment 부분갱신으로 사용성 핵심(깜빡임·실시간 숫자)은 이미 80% 달성.
Next.js는 남은 20%(네이티브 앱 수준 반응성)를 위해 수 배 비용이 드는 큰 투자 →
**현 단계에선 보류**, 실사용 트래픽/요구가 커지면 재검토.
